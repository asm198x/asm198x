Grounding Context

Codebase context. Asm198x is a Rust workspace of three crates: isa (19 CPU encoding specs as &'static data), isa-disasm, and the asm198x engine/CLI. Every CPU is validated byte-identical against a real reference assembler (acme, ca65, pasmo, sjasmplus, lwasm, vasm, rgbasm, asl), with round-trips and ~250k-encoding opcode sweeps. The release binary is 2.0 MB for all 19 CPUs. This run's evidence layer was the same-day four-agent quality assessment (engine/dialects, ISA spec crate, disassembler, CLI/tests/docs/CI) with file:line findings, plus the issue-tracker history and the 198x family map (Emu198x emulator, Forge198x IDE, Code198x curriculum, Build198x media tools).

External context (fresh web research, value: high). Three converging findings shaped the survivors: (1) no cross-assembler debug-info standard exists in the retro space — sjasmplus SLD and ca65 .dbg are single-tool formats each emulator imports bespoke; (2) local-label scoping is the most-praised assembler feature and macros are "the escape hatch users reach for"; (3) no retro LSP or browser playground is driven by a validated ISA spec — all incumbents hand-roll encoders. Community trust patterns: scene fragmentation is chronic, DOSBox-hosted tools are distrusted, and docs/banking-model clarity earns as much trust as features. Machine-readable-diagnostics (rustc JSON) and published-spec ecosystems (riscv-opcodes, sail-riscv) are the transferable cross-domain patterns.

Binding constraints honored. Source-compatible dialects (no invented syntax); single binary + subcommands; fused linking; new-CPU validatability gate; no LLVM; sibling ownership boundaries (Build198x masters media, Play198x renders, Emu198x executes).

1

One debug-info format for every CPU — the DWARF of retro

axis: debug-metadata Confidence 85% Complexity: Medium
Description
Define one machine-readable debug-info schema — line↔address mapping with SLD-style "long addresses" (address + bank/page so banked memory disambiguates), symbols, scopes, sections — emitted by the shared two-pass engine via a single --debug= flag, identically for all 19+ CPUs. Ship compatibility exporters (sjasmplus SLD for DeZog/CSpect, ca65 .dbg for Mesen2) so existing debuggers work on day one, while Emu198x and Forge198x consume the native schema. The existing listing/symbol/map gap becomes the on-ramp: --sym/--listing are the first two renderings of the same data.
Basis
external: research digest — "there is NO cross-assembler DWARF-equivalent in retro space — every assembler grew its own format, every emulator wrote its own importer." direct: Assembly.symbols: BTreeMap<String, i64> already computed for every CPU (engine.rs:28) and exposed nowhere (verifier-confirmed).
Rationale
This is the strongest whitespace the research found, and Asm198x is structurally the only project that can fill it: incumbents are single-CPU community tools, so none can define a cross-CPU format — while the 198x family owns the emitter (one engine, 19 CPUs), the consumer (Emu198x debug views), and the IDE (Forge198x) in one house. Source-level debugging is what separates "a good assembler" from "a platform."
Downsides
A format design is a long-term compatibility commitment — worth one deliberate design session (and a decision record) before any emitter code. Banked/long-address semantics vary per machine and need care. Adoption outside the family depends on the compat exporters actually working with DeZog/Mesen2.
The structural advantage: incumbents each own one arrow; the 198x family owns the whole right-hand picture.
2

The verdict pipeline — arbiter escrow → public conformance ledger

axis: correctness-product Confidence 90% Complexity: Medium
Description
Snapshot every reference-assembler verdict — source → bytes, accept/reject, tool name + exact version — into a committed, versioned golden-verdict corpus. CI and contributors then validate against the corpus with zero tools installed; the live references are needed only to grow it. On top of the corpus, publish a per-release conformance ledger (per CPU: reference tool + version, case count, corpus hash, pass rate — the wpt.fyi / Certificate-Transparency pattern) on the docs site, and extend the corpus with negative conformance: programs the references reject or truncate, with asserted Asm198x behaviour and a published divergence classification (we-accept/they-reject, they-accept/we-reject, both-accept-different-bytes).
Basis
direct: "Reference-arbitrated suites are all #[ignore]d; CI installs no reference assemblers — the core guarantee runs only on the maintainer's machine" (verified this session); the full reference-tool set exists on exactly one machine. direct: issues #36 and #33 show real-world bugs concentrate at the acceptance boundary, which byte-identical happy-path testing is blind to by construction. external: wpt.fyi, Certificate Transparency, reproducible-builds rebuilder networks.
Rationale
The project's actual moat — byte-identical validation against 8+ references — is currently invisible, unverifiable, and bus-factor-one. One system fixes three things: CI can finally enforce the core guarantee (escrow), skeptical adopters get a receipt instead of a README claim (ledger), and the aging arbiters' behaviour is preserved — which is the project's own rescue mission applied to its trust chain. This sequencing beat the standalone dashboard idea precisely because the corpus must exist before there is anything to publish.
Downsides
Corpus curation and format design take real effort; verdict snapshots must record tool versions rigorously (which also lays the groundwork for dialect-edition pinning later). Repo size growth needs managing (verdicts compress well; store distilled forms).
Live tools only grow the corpus; everything downstream runs from the committed verdicts.
3

One contract, four surfaces — API + diagnostics core; LSP, MCP, WASM as renderers

axis: family-integration Confidence 85% Complexity: High (staged)
Description
Establish one structured core contract — assemble(cpu, dialect, source) → Result<Assembly, Diagnostics> with span-carrying, stable-coded, JSON-serializable diagnostics (rustc's "mechanically applicable fix" flag included), a public Dialect trait, and a spec-query API — then ship the user-facing surfaces as thin renderers over it: asm198x lsp (hover = cycles/flags/encoding from the spec itself), asm198x mcp (session-stateful agent surface, matching Emu198x's MCP-first family convention), and a WASM browser playground with byte-identical permalinks embedded in Code198x lesson pages. All are subcommands of the single binary, per the packaging decision.
Basis
direct: ~26 flat assemble_* functions with 3 return shapes, pub(crate) trait Dialect (dialect.rs:25, verifier-confirmed), span-less AsmError — Forge198x is "gated on Asm198x maturity" and literally cannot integrate today. external: no retro LSP is spec-driven and no playground carries a validated encoder (both confirmed whitespace); rustc JSON diagnostics as the enabling pattern.
Rationale
The verifier called this the strongest strategic call in the set: shipping LSP/MCP/WASM before stabilizing the contract means redoing surface work three times. Done in order, one investment unblocks the IDE sibling, the agent-native goal, the classroom (zero-install assembler in every lesson page — the single biggest beginner drop-off removed), and modern editor UX — and every future CPU inherits all four surfaces automatically.
Downsides
The largest total scope in the set — it only works as a staged programme (contract → JSON diagnostics → one surface at a time). Committing the public API is a semver responsibility (#[non_exhaustive] from day one). LSP incrementality questions can be deferred but not forever.
Four thin renderers, one contract — the sequencing is the idea.
4

One macro engine, N dialect skins

axis: language-surface Confidence 80% Complexity: High
Description
Implement macro expansion, conditional assembly, include/incbin, and local-label scoping once at the engine layer, then give each dialect a thin syntax-mapping skin — sjasmplus spells it MACRO/ENDM + DUP, ca65 spells it .macro/.endmacro + .repeat, rgbasm its own way — each skin differentially validated against its reference assembler, exactly like the instruction surface already is.
Basis
direct: zero macros/conditionals/includes exist today; sjasmplus.rs:16 carries a literal TODO ("modules, macros, DUP"); the only local-label support is an ad-hoc private hack (qualify_locals, z80.rs:455–496). external: the research digest's two loudest community signals — local-label scoping is "the single most-repeated praise pattern" and macros are "the escape hatch users reach for when a feature is missing."
Rationale
This is the largest gap between "byte-identical validated" and "usable for a real, non-trivial program" — curriculum-sized sources survive without macros; serious homebrew and demoscene work hits the wall immediately. Building it per-dialect would multiply the already-measured 16–20× dialect copy-paste problem by the hardest feature in the codebase; the engine-level architecture decision must precede the first line of macro code.
Downsides
The verifier's honest caveat: "N thin skins" undersells how much macro semantics (not just spelling) differ across ca65/sjasmplus/rgbasm — expect a shared core plus real per-dialect semantic work, validated dialect-by-dialect. This is the project's biggest single language-surface investment.

Cycle-budget analyzer & cycle-honest listing

axis: debug-metadata Confidence 75% Complexity: Medium
Description
Emit listings (human table and machine JSON) annotating every line with size and cycle cost — min/max with branch-taken and page-cross penalties — plus per-block and per-label totals, sourced from the spec's validated Cycles data. Add a source-level assertion that fails the build when a routine exceeds its budget — the raster programmer's contract, checked by the assembler:
Basis
direct: the Form-model specs already carry Cycles { base, page_cross, branch_taken }, machine-checked in tests, and used for nothing; no listing output exists at all. external: llvm-mca (static machine-code timing) and WCET analysis from real-time systems — the same mechanism, pointed at CPUs where cycle-exactness is the defining discipline (racing the beam, stable rasters).
Rationale
Demoscene and game programmers count cycles by hand today; no retro assembler does this uniformly across CPUs. It converts the spec's least-exploited asset into the feature that makes expert users switch — and gives Code198x a way to teach timing with tool support instead of folklore.
Downsides
The honest hidden cost: cycles are absent from all six bespoke spec models (m68k, 6809, 65816-ext, PDP-11, TMS9900, Z8000), so full coverage forces the backfill — which is also exactly the forcing function that upgrades the spec from "documentation-grade" to load-bearing. 68000-class timing (EA-dependent) is genuinely harder than 6502-class.
; asm198x: cycles(irq_raster) <= 224        ← build fails if exceeded
irq_raster:
    lda $d019     ; 4      [4]
    sta $d019     ; 4      [8]
    ldx offset    ; 4      [12]
    lda sine,x    ; 4+1p   [16..17]   p = page-cross penalty
    sta $d020     ; 4      [20..21]
    bne .loop     ; 2+1t   [22..24]   t = taken
                  ; ── block total: 22 min / 24 max ──
Sketch of a cycle-honest listing: every line priced, every block totalled, budgets enforced.
6

Dialect-to-dialect source converter — asm198x convert

axis: language-surface Confidence 65% Complexity: Medium-High
Description
Because every dialect front-end already normalises to the same internal operations, emit that representation back out as a different existing dialect of the same CPU: asm198x convert --from pasmo --to sjasmplus prog.asm. Correctness is self-verifying — assemble both sides, diff the bytes — and no syntax is invented, so the source-compatible stance holds: both ends are real reference dialects.
Basis
direct: the shared normal form exists (one Expr evaluator, the Operation::Encoded seam) with N front-ends over it. external: scene fragmentation is chronic (demoscene→sjasmplus, homebrew→WLA-DX, ROM-hacking→xkas) and decades of source are trapped in dying tools' dialects; Python 2to3/jscodeshift are the mechanism precedent, with the byte-diff as the built-in snapshot test.
Rationale
This is "rescue over replace" applied to other people's code, not just toolchains — the migration path that turns orphaned-source communities into users, offering something no single-dialect incumbent can attempt. Its self-verification doubles as a continuous cross-dialect consistency audit of the front-ends.
Downsides
Round-trip fidelity is the open question: comments, formatting, and (once they exist) macros need carrying through the IR, which today preserves none of them. Scope it as instruction/directive-level conversion first, fidelity later. Lowest confidence of the survivors — but the highest "nobody else can even try" score.
The converter's proof obligation is the tool's own core competence: byte-identical assembly.
7

The self-verifying docs site

axis: dx Confidence 90% Complexity: Medium
Description
Build the desired docs site so that it cannot drift: per-CPU instruction references generated from the isa crate (opcode, layout, cycles, flags, provenance link into the umbrella reference/ library); per-dialect directive-support matrices derived from the conformance corpus; every code sample on the site assembled by the real binary in CI; diagnostic explain-pages keyed by stable error codes (shellcheck-wiki style) that teach the period-hardware why — branch range, page-cross, banked memory — citing datasheets. The conformance ledger (idea 2) publishes here. Prose that can rot is minimised; generated data that cannot is maximised.
Basis
direct: the drift already happened — README says "Five CPUs" against 19 shipped, the docs repo holds one dialect page and a stale spec-format doc (verified this session). external: WLA-DX's cautionary tale that docs/banking-model confusion is resented as much as bugs; shellcheck's coded-wiki pattern; the digest's finding that docs clarity is trust in this community.
Rationale
The naive docs site (hand-written pages) creates a second source of truth that is provably already stale at 19 CPUs and will lose harder at 30. Generating from the spec makes the docs site a correctness artifact with the same trust property as the assembler — and it's the natural venue for the ledger, the explain-pages, and the spec's public face. mdBook + the existing Vale/House198x lint (promoted to a CI gate) fits the boring-tech stance; every future CPU documents itself the moment its spec lands.
Downsides
Needs a spec-export/query layer to generate from — which is the same layer the LSP and Emu198x validation want (build once, feed three consumers). Generated tables still need hand-written prose around them (directives, worked examples); the generator only removes the drift-prone half.
